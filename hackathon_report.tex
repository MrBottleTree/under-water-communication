% SECTION 2 --- Abstract
\section{Abstract}
This report presents an Android-to-Android underwater communication system that uses only a phone flashlight for transmission and a phone camera for reception. The final design avoids sustained ON/OFF signaling and instead uses ON-duration encoding with two timing families (Type 1 and Type 2), so the receiver can classify bits from edge transitions at 20 FPS. The receiver pipeline combines ROI-based grid analysis, rolling-window histograms, Otsu thresholding, confidence-weighted voting, and ON-duration decoding. On top of this physical layer, we implement a protocol with calibration constants, packet request/response flow, CRC-8 checks, and a role-based finite state machine. A dedicated test UI supports manual signal generation, full protocol runs, and live tuning of threshold and grid parameters. The system currently supports fixed 5-bit character encoding with packets of five characters and is designed to be extended to richer alphabets and larger payloads.

% SECTION 3 --- Problem Statement
\section{Problem Statement}
The core challenge is to enable reliable smartphone-to-smartphone communication underwater without any external hardware. The transmitter can only use the built-in flashlight and the receiver can only use camera frames. This means the link must work under strong constraints: limited frame rate, torch latency, sensor noise, changing ambient light, and motion from hand-held devices.

Underwater conditions make this harder than in-air optical signaling. Scattering and reflections blur the bright source and reduce contrast between ON and OFF states. Background intensity changes can happen within a few frames, especially when the camera auto-exposure is active. For this reason, the system must lock the camera pipeline to manual control and treat signal transitions as the main source of information.

The implementation targets 20 FPS with a 480p analysis stream, because this level is practical for real-time processing on mobile devices while keeping enough temporal resolution for millisecond-scale pulse classes. We also restrict ourselves to no external sync clock, no external light sources, and no added optics. All protocol reliability must come from software design, timing margins, and error checks.

Reliability goals are defined at two levels. At bit level, the receiver must keep the bit error rate (BER) low enough that packets remain decodable after CRC checks. At message level, the goal is stable packet decoding success with retransmission support, even when occasional edges are missed. These goals drive both the physical-layer timing design and the protocol state machine.

\subsection{High-Level Design Explanation}
The architecture separates responsibilities cleanly. The camera control layer stabilizes frame timing and exposure behavior. ROI detection and grid analysis convert each frame into a compact signal estimate. The decoder then converts edge transitions into bits, while protocol modules convert those bits into constants, packet requests, and payloads.

This separation keeps implementation and debugging manageable in a single-activity Android app. Physical-layer tuning (grid size, thresholds, window size, zoom) can be changed without rewriting packet logic. In the same way, protocol retries and role switching can be changed without touching image processing code.

The current UI-level payload entry supports messages up to 80 characters for interactive testing. Character mapping currently focuses on lowercase alphabets plus limited punctuation and space. Because encoding and protocol are modular, the design can be extended to richer symbol sets and larger packet formats without changing the core detection loop.

% SECTION 5 --- ROI Detection Algorithm
\section{ROI Detection Algorithm}
The receiver operates at 20 FPS on a 480p frame stream. Each frame is divided into grid cells, and each cell maintains a brightness history. During tuning, different cluster granularities were evaluated (1x1, 2x2, 4x4, 8x8 style aggregation) to balance spatial precision and noise immunity. In the final pipeline, connected high-confidence cells form candidate clusters, and the dominant cluster feeds the bit vote.

For each grid, the algorithm updates a brightness histogram and applies Otsu thresholding. If the histogram has a clear twin-peak structure (dark and bright modes), inter-class variance is high and confidence rises. If the grid sees mostly one level, confidence drops. This confidence-weighted design prevents random bright regions from dominating the result.

The bit decision is not based on sustained brightness. Instead, the system tracks transitions. Sustained ON fails because a stable bright scene collapses the histogram into one peak, so Otsu confidence approaches zero and bitVote becomes weak. Sustained OFF has the same issue. This is why every useful symbol must contain ON/OFF transitions.

Rolling-window histograms with \(N=3\) are used by default because they provide fast adaptation while preserving clear transition spikes. With three frames, the system still sees mixed states during edges, but quickly settles to adapted states where vote returns near zero. This makes edge timing easier and reduces false latching in steady light.

\begin{table}[H]
\centering
\caption{Main ROI and decoder hyperparameters used during tuning.}

\toprule

Parameter tuning is environment-dependent. In daylight, lower exposure and moderate confidence threshold usually work best. In low light, slightly larger grids and lower confidence thresholds help maintain sensitivity. Under flickering backgrounds, raising confidence threshold and validating cluster continuity reduces false edges. Under motion, larger grids and stable vote threshold values improve robustness.

Digital zoom (1x, 2x, 4x) is a key practical control. Zoom improves signal-to-noise ratio (SNR) when the transmitter is far away because more of the frame budget is assigned to the active light region. At short range in stable conditions, 1x is usually sufficient. At longer range or stronger scattering, 2x or 4x often improves transition visibility and reduces missed edges.

\subsection{Why ON-Duration Encoding and 100 ms Margins}
ON-duration encoding works because the receiver only needs accurate RISE and FALL edge times, not absolute intensity levels. This is robust to moderate brightness drift and underwater scattering. Timing clusters are separated by 100 ms margins between class boundaries, which is two frame periods at 20 FPS. That gap is large enough to avoid overlap under normal jitter.

Packet payload uses fixed 5-bit encoding. This keeps decoding deterministic and timing-predictable, which is important for ON-duration links.

\subsection{Exploration: Custom Balanced Huffman Encoding (Discarded Idea)}
We explored a custom balanced Huffman map to reduce average bits per character. In static simulations, this reduced expected payload length. However, variable-length symbols created variable timing at the link layer, which increased ambiguity in edge segmentation and packet boundary handling.

Because this system depends on clear timing classes, the variable-length design made real-time decoding and retransmission logic more complex than expected. We therefore kept fixed 5-bit symbols for the final system.

% SECTION 8 --- Packet Structure
\section{Packet Structure}
Each packet carries exactly five characters. With 5-bit fixed encoding, that gives 25 data bits. An 8-bit CRC-8 field is appended, so each packet has 33 bits total.

\begin{figure}[H]
\centering
\begin{tikzpicture}[x=0.24cm,y=0.95cm,font=\small]
\draw[thick] (0,0) rectangle (33,1);
\draw[thick] (5,0) -- (5,1);
\draw[thick] (10,0) -- (10,1);
\draw[thick] (15,0) -- (15,1);
\draw[thick] (20,0) -- (20,1);
\draw[thick] (25,0) -- (25,1);
\node at (2.5,0.5) {C1};
\node at (7.5,0.5) {C2};
\node at (12.5,0.5) {C3};
\node at (17.5,0.5) {C4};
\node at (22.5,0.5) {C5};
\node at (29,0.5) {CRC-8};
\node[below] at (12.5,0) {25 data bits};
\node[below] at (29,0) {8 bits};
\end{tikzpicture}
\caption{Packet bit layout: 5 characters + CRC-8.}
\end{figure}

All packet bits are transmitted using Type 1 timing. Since each T1 bit is 600 ms, one packet takes \(33 \times 600\) ms \(= 19.8\) seconds.

PAD is used when the source string length is not a multiple of 5 characters. Example: if the user sends \texttt{"hi"}, the packet payload becomes \texttt{"hi"} + three PAD characters so the bitstream always has 25 data bits.

Example 5-character payload: \texttt{"hi, ?"}.
\begin{center}
\texttt{h = 00111,\ i = 01000,\ , = 11010,\ [space] = 11110,\ ? = 11101}
\end{center}
These 25 data bits are followed by CRC-8 bits before transmission.

CRC-8 is used as a packet integrity check (polynomial \(0x07\), CRC8-CCITT). A single-bit error will almost always alter the computed CRC value.

\begin{table}[H]
\centering
\caption{Example CRC mismatch behavior for one-bit corruption.}
\begin{tabular}{ll}
\toprule
Transmitted packet & Data bits \(D\) + CRC \(C\) \\
Received packet & Data bits \(D'\) (one bit flipped) + same CRC \(C\) \\
Receiver check & Recomputed CRC \(C' \neq C\), packet rejected and retransmitted \\
\bottomrule
\end{tabular}
\end{table}

\subsection*{Q:no Special Case}
\texttt{Q:no} is a mixed-format frame:
\begin{itemize}
\item \texttt{Q} is transmitted as 3 T2 bits (\(5400\) ms).
\item The packet number \texttt{no} is transmitted as 4 T1 bits (\(2400\) ms).
\end{itemize}
Total duration is \(5400 + 2400 = 7800\) ms.

Example for requesting packet 3:
\begin{center}
\texttt{Q (110 in T2)} + \texttt{0011 (packet no in T1)}
\end{center}
Using T2 for the constant prefix improves detection stability, while T1 for the packet index keeps request messages shorter.

The FSM coordinates two roles. The responder path begins after receiving \texttt{C1}, then enters \texttt{READY\_SEND}. The initiator path begins after receiving \texttt{C2}, then enters \texttt{REQUESTING}. This role split avoids both peers trying to send payloads at the same time.

During \texttt{REQUESTING}, the node asks for packet numbers sequentially using \texttt{Q:no}. During \texttt{SENDING}, the node responds with either \texttt{SPE} (start + packet + end) or a bare \texttt{E} when packet range is exhausted. This keeps packet transfer explicit and easy to recover after loss.

\texttt{RESPONSE\_DELAY\_MS} (3000 ms) is critical. After each TX completion, the FSM waits before opening the next RX window. This prevents TX/RX overlap and gives the remote side enough time to finish torch transitions.

% SECTION 12 --- What Actually Works? / Final Deliverables
\section{What Actually Works? / Final Deliverables}
The full protocol stack in this report captures everything we attempted during the hackathon, including calibration, role switching, packet request flow, and end-to-end FSM orchestration. We made strong progress on these blocks, but the complete automated protocol loop was not fully stabilized within the hackathon time frame. We are continuing work to complete and harden this path.

The part that works reliably today is manual packet transmission and manual packet reception. Using the test UI, we can manually transmit the packet signal (\texttt{P}) with a chosen message from the fixed transmitter device. On the receiver side, the bitstream is decoded correctly and mapped back into the corresponding 5-character payload.

This behavior is consistent with the current \texttt{SignalProtocol} implementation:
\begin{itemize}
\item \texttt{encodePacket()} constructs a 33-bit packet (25 data bits + 8 CRC bits) using Type 1 timing.
\item \texttt{checkPacketCrc()} validates integrity at the receiver.
\item \texttt{decodePacket()} converts validated bits back into the 5-character decoded output.
\end{itemize}

Therefore, the final deliverable we are submitting is a working manual underwater optical link for fixed-position transmitter and receiver devices, with reliable transfer of 5 characters at a time.

\subsection{Evaluation}
We performed a simple validation run with approximately 40 manually transmitted 5-character packet sequences. In this run, all sent messages were decoded fully and correctly at the receiver. For this evaluated set, the observed packet decoding outcome implies an effective BER of 0.

\section{Technical Stack}
The implementation is built in Kotlin for Android and uses CameraX 1.4.0 with Camera2Interop for manual camera control. The code intentionally avoids MVVM and dependency injection frameworks (Dagger/Hilt) to keep low-level timing behavior and UI wiring in one place during rapid protocol iteration.

This stack is intentionally minimal but complete: enough control for timing-critical behavior without introducing framework overhead that can hide latency and state transitions.

\section{Conclusion}
This project delivers a practical underwater optical communication stack using only smartphone hardware. The final system combines transition-driven ROI decoding, ON-duration bit encoding, and a role-based protocol FSM with CRC-backed retransmission.

The key reliability gain comes from matching the physical layer to camera realities: fixed 20 FPS timing, AE-off capture, clear ON-duration clusters, and transition-based detection instead of sustained brightness detection. On top of this, protocol-level checks and retries convert noisy bitstreams into usable packet delivery.

Future work will focus on in-water parameter retuning, BER characterization across distance/turbidity ranges, adaptive threshold presets, and larger payload support while preserving real-time decode stability.
